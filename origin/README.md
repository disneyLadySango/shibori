# Origin

`/origin/`はShiboriの意図と制約の唯一の正である。各文書は「初期」「現在」といった曖昧な名前で上書きせず、識別子と状態を持つ。

## 4層

1. `1-language/`: 下位層へ配給する語彙
2. `2-stories/`: Who / Why / ペインと、その根拠・履歴
3. `3-ac/`: 採用中または検討中の具体的な賭けに対する成果基準
4. `4-test-specs/`: ACの観測方法

## 構造上の不変条件

1. ユビキタス言語の正本は`1-language/glossary.md`一つ。可読性のため分割しても、単一の索引から一つの語彙体系として配給する。
2. 全ユーザーストーリーを統括するマップは`2-stories/story-map.md`一つ。
3. 発見したユーザーストーリーは、選択前にすべて`2-stories/backlog/US-xxx.md`へ置く。
4. 選択したユーザーストーリーだけを`active/`へ移し、終了後も`archive/`へ履歴を残す。
5. 各ユーザーストーリーは`3-ac/{backlog|active|archive}/US-xxx/`に一つ以上のACを持つ。
6. 各ACは第4層に複数のテスト仕様を持てる。テスト仕様はHowとともに増え、backlogとして管理される。

## 文書状態

| 状態 | 意味 |
| --- | --- |
| `backlog` | 発見済みだが、現在の課題解決対象として未選択 |
| `active` | 人間が現在の課題解決対象として選択済み |
| `proposed` | 人間の確認前。実装根拠にできない |
| `approved` | 人間が意図として承認済み |
| `superseded` | 別文書へ置き換え済み。履歴として残す |
| `retired` | 課題が偽と判明した、または撤退条件に達した |

状態変更はPRで行い、`approved`へ変える際は承認者と日付を記録する。内容変更が意味を変える場合、同じファイルを黙って上書きせず`revision`を増やし、変更理由を末尾へ追記する。

## 履歴規則

- 第2層のペインはACが失敗しても削除しない。
- 観測事実は`2-stories/evidence/`へ、課題は`2-stories/{backlog|active|archive}/`へ分ける。
- 失敗した具体策と学びは`2-stories/history/`へ残す。
- 第3層の終了したACは、結果を追記して`3-ac/archive/`へ移す。
- 第4層も対応するACと同時に`4-test-specs/archive/`へ移す。
- archiveは不変ではないが、事実誤認の訂正以外では書き換えず、追記を使う。

## 依存方向

`1-language → 2-stories → 3-ac → 4-test-specs → 射影`

上位層の意味変更時は、下位文書の参照IDと射影への影響を監査する。
